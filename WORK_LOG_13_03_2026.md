## Задача 1: Поиск треков и альбомов

### Проблема

Поиск не отображал результаты несмотря на то, что бэкенд API работал корректно. Пользователь вводил название альбома или исполнителя — получал «No results found».

### Диагностика

Проверка через curl из Docker-контейнера показала: бэкенд возвращает результаты корректно. Проблема была в связке фронтенда и nginx:

1. **Глобальный `isLoading`** — `SearchPage` использовал `isLoading` из `useMusicStore`. Когда левая панель (`LeftSidebar`) одновременно загружала список альбомов (`fetchAlbums()`), флаг `isLoading = true` скрывал результаты поиска и показывал «Searching...» бесконечно.

2. **307 Redirect** — запрос `/api/search?q=...` (без trailing slash) вызывал 307-редирект FastAPI на `/api/search/?q=...`. Nginx вставлял в заголовок `Location` внутренний Docker-адрес (`http://172.17.x.x:8000/...`), недоступный из браузера — поиск молча завершался с ошибкой.

3. **Поиск не охватывал `album_name`** — треки искались только по `title` и `artist`, но не по названию альбома.

### Решение

**Бэкенд** (`app/routes/search.py`):
```python
# Двойной роут — убирает 307-редирект
@router.get("")
@router.get("/")
def search(q: str = Query(..., min_length=1), db: Session = Depends(get_db)):
    ...
    # Добавлен поиск по album_name
    tracks = db.query(Track).filter(
        or_(
            Track.title.ilike(f"%{query}%"),
            Track.artist.ilike(f"%{query}%"),
            Track.album_name.ilike(f"%{query}%"),   # ← новое
        )
    ).limit(20).all()
```

**Фронтенд** (`SearchPage.tsx`) — полная переработка:

```typescript
// БЫЛО: shared isLoading из store → конфликт с fetchAlbums()
const { search, isLoading } = useMusicStore();

// СТАЛО: локальный isSearching — изолирован от других операций
const [isSearching, setIsSearching] = useState(false);

const doSearch = async (q: string) => {
    setIsSearching(true);
    try {
        // URL со слэшем — без 307-редиректа
        const response = await axiosInstance.get("/search/", {
            params: { q: q.trim() },
        });
        setResults(response.data);
    } finally {
        setIsSearching(false);
    }
};
```

**Debounce** — автопоиск при вводе с задержкой 400 мс:
```typescript
const handleQueryChange = (value: string) => {
    setQuery(value);
    if (debounceTimer.current) clearTimeout(debounceTimer.current);
    debounceTimer.current = setTimeout(() => doSearch(value), 400);
};
```

**Начальный экран** — вместо простых жанровых кнопок теперь отображаются все альбомы из базы, что позволяет перейти к нужному контенту сразу без ввода запроса.

### Результат

- Поиск работает без задержки — результаты появляются через 400 мс после ввода
- Треки находятся по названию трека, имени исполнителя и названию альбома
- Альбомы находятся по названию и исполнителю
- Клик по альбому в результатах → открывается страница альбома со всеми треками
- Клик по треку → запускается воспроизведение

## Задача 2: Your Library — Моя библиотека

### Проблема

Раздел «Your Library» был пустым. Вкладка «Albums» показывала все альбомы из базы данных (без фильтрации), кнопка «Create playlist» не работала. Пользователь не мог сохранить альбомы или треки в «свою» библиотеку.

### Решение

**Новый Zustand store** (`useLibraryStore.ts`) с сохранением в `localStorage`:

```typescript
import { create } from "zustand";
import { persist } from "zustand/middleware";

export const useLibraryStore = create<LibraryStore>()(
    persist(
        (set, get) => ({
            savedAlbums: [],
            likedSongs: [],

            saveAlbum: (album) =>
                set((s) => ({
                    savedAlbums: s.savedAlbums.some((a) => a.id === album.id)
                        ? s.savedAlbums
                        : [album, ...s.savedAlbums],
                })),
            removeAlbum: (id) =>
                set((s) => ({ savedAlbums: s.savedAlbums.filter((a) => a.id !== id) })),
            isAlbumSaved: (id) => get().savedAlbums.some((a) => a.id === id),

            likeSong: (song) =>
                set((s) => ({
                    likedSongs: s.likedSongs.some((t) => t.id === song.id)
                        ? s.likedSongs
                        : [song, ...s.likedSongs],
                })),
            unlikeSong: (id) =>
                set((s) => ({ likedSongs: s.likedSongs.filter((t) => t.id !== id) })),
            isSongLiked: (id) => get().likedSongs.some((t) => t.id === id),
        }),
        { name: "spotify-library" }  // ключ в localStorage
    )
);
```

**Обновлённый `LibraryPage.tsx`**:

| Вкладка | Содержимое | Действия |
|---------|-----------|---------|
| Albums | Только сохранённые пользователем альбомы | Клик → страница альбома; кнопка ♥ «Saved» убирает из библиотеки |
| Liked Songs | Любимые треки | Клик → воспроизведение; кнопка ♥ убирает из списка |

При пустой библиотеке — информативное сообщение с подсказкой как добавить контент.

### Почему localStorage, а не база данных

Для реализации через БД потребовалось бы:
- Новая таблица `user_saved_albums`
- Новые API-эндпоинты (GET/POST/DELETE)
- Авторизация для каждого запроса

localStorage — pragmatic решение для учебного проекта: данные сохраняются между сессиями в рамках одного браузера, не требуют серверной инфраструктуры.

## Задача 4: Исправление критического бага поиска

### Проблема: 307 Temporary Redirect

**Симптом:** пользователь вводит запрос → «No results found»

**Цепочка событий:**
```
1. Браузер: GET /api/search?q=nech
2. Nginx: проксирует на http://backend:8000/api/search?q=nech
3. FastAPI (redirect_slashes=True по умолчанию):
   → 307 Redirect → Location: http://backend:8000/api/search/?q=nech
4. Nginx: пересылает 307 в браузер с внутренним Docker-адресом
5. Браузер: пытается открыть http://backend:8000/... → ERR_CONNECTION_REFUSED
6. axios catch: setResults({tracks:[], albums:[], artists:[]})
7. UI: "No results found"
```

Та же проблема ранее была с `POST /api/songs` (решена в предыдущей сессии).

**Решение (двустороннее):**

```python
# app/routes/search.py — принимаем оба варианта URL
@router.get("")   # ← без слэша
@router.get("/")  # ← со слэшем
def search(...):
```

```typescript
// SearchPage.tsx — явно используем URL со слэшем
const response = await axiosInstance.get("/search/", { params: { q: q.trim() } });
```

**Проверка через PowerShell:**
```powershell
Invoke-WebRequest "http://localhost:3000/api/search/?q=nech"
# StatusCode: 200
# {"tracks":[...], "albums":[...], "artists":[]}
```

## Технические решения и их обоснование

### 1. Изоляция состояния загрузки поиска

**Проблема:** общий `isLoading` в `useMusicStore` использовался всеми компонентами (LeftSidebar, HomePage, SearchPage). Любая операция установки `isLoading: true` в одном компоненте влияла на UI другого.

**Решение:** каждый компонент, требующий независимой индикации загрузки, должен иметь собственный локальный state (`useState`). Глобальный `isLoading` в store — антипаттерн для параллельных операций.

### 2. Zustand persist для библиотеки

**Выбор `zustand/middleware persist`** — данные сохраняются в `localStorage` автоматически при каждом изменении state. При перезагрузке страницы store восстанавливается из `localStorage`. Это требует нулевого кода на сервере и идеально подходит для пользовательских предпочтений.

**Потенциальная проблема:** если данные в БД изменятся (например, альбом будет удалён), localStorage будет хранить устаревший объект. В production-системе потребовалась бы синхронизация с сервером.

### 3. Двойной декоратор для FastAPI роутов

**Правило:** любой роут в FastAPI, доступный из браузера через Nginx reverse proxy, должен принимать URL как со слэшем, так и без:

```python
@router.get("")
@router.get("/")
def endpoint(...):
```

Альтернатива — отключить `redirect_slashes` глобально:
```python
app = FastAPI(redirect_slashes=False)
```

Или настроить Nginx, чтобы он автоматически добавлял слэш. Метод с двойным декоратором — наиболее явный и не требует правки nginx.conf.

### 4. Debounce в поиске

400 мс — баланс между отзывчивостью и количеством запросов:
- < 200 мс: слишком много запросов при быстром вводе
- > 600 мс: ощутимая задержка для пользователя
- 400 мс: стандарт для поиска в реальном времени
