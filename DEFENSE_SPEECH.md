## 1. Цель и идея проекта

Цель проекта — создать реальное fullstack-приложение с нуля, применяя технологии, которые используются в современной коммерческой разработке.

Приложение позволяет пользователям:
- Слушать музыку — воспроизводить треки, управлять очередью, регулировать громкость и перематывать
- Искать треки, альбомы и исполнителей с мгновенным автопоиском
- Сохранять понравившиеся альбомы и треки в личную библиотеку
- Просматривать информацию об исполнителях
- Создавать плейлисты и управлять ими
- Регистрироваться и входить в систему с защищённой аутентификацией
- Управлять контентом через отдельную **админ-панель**

## 2. Технологический стек

Проект реализован как **трёхзвенная архитектура**:

### Backend — FastAPI (Python)
- **FastAPI 0.109+** — современный асинхронный фреймворк для REST API
- **SQLAlchemy 2.0** — ORM для работы с базой данных
- **Alembic** — управление миграциями схемы базы данных
- **PostgreSQL 16** — реляционная СУБД для хранения данных
- **python-jose + bcrypt** — JWT-аутентификация и хеширование паролей
- **Uvicorn** — ASGI-сервер для запуска приложения

### Frontend — React + TypeScript
- **React 19 + TypeScript 5** — компонентный UI с типизацией
- **Vite 7** — быстрый сборщик проекта
- **Tailwind CSS 3** — утилитарные стили, адаптивный дизайн
- **Zustand 5** — лёгкое управление глобальным состоянием
- **React Router 7** — клиентская маршрутизация
- **Radix UI** — доступные UI-компоненты
- **Axios** — HTTP-клиент с interceptor'ами

### Инфраструктура
- **Docker + Docker Compose** — контейнеризация всех сервисов
- **Nginx** — прокси-сервер, раздача статики и медиафайлов

## 3. Архитектура приложения

```
Браузер → Nginx (port 3000)
              ├── /         → React SPA (статические файлы)
              ├── /api/*    → FastAPI backend (port 8000)
              └── /media/*  → FastAPI (загруженные медиафайлы)
```

Все три сервиса — **frontend**, **backend**, **postgresql** — запускаются через Docker Compose одной командой:

```
docker compose up --build -d
```

## 4. База данных

Схема базы данных состоит из четырёх основных таблиц:

| Таблица | Назначение |
|---------|-----------|
| `auth_users` | Аккаунты пользователей (email, bcrypt-хеш пароля) |
| `user_profiles` | Профили (имя, аватар, биография) |
| `albums` | Альбомы (название, исполнитель, обложка, год) |
| `tracks` | Треки (название, исполнитель, длительность, аудиофайл, FK на альбом) |

Связь между таблицами организована через **внешние ключи** (Foreign Key). При удалении альбома — связь трека обнуляется (`SET NULL`), а не удаляется трек — это намеренное архитектурное решение для сохранности данных.

Миграции схемы управляются через **Alembic** — при изменении модели создаётся файл миграции, который применяется командой `alembic upgrade head`.

## 5. Аутентификация

Реализована **JWT-аутентификация**:

1. Пользователь вводит email и пароль
2. Backend проверяет пароль через **bcrypt**-сравнение хешей
3. При успехе выдаётся **JWT-токен** (подписан секретным ключом)
4. Токен хранится в localStorage на фронтенде
5. Каждый защищённый запрос отправляется с заголовком `Authorization: Bearer <token>`
6. При истечении токена **Axios interceptor** автоматически обновляет сессию — пользователь не замечает разрыва

## 6. Ключевые технические решения

### 6.1. Управление состоянием — Zustand

Вместо стандартного React Context использована библиотека **Zustand** — лёгкая, без лишнего кода.

В проекте реализовано 5 store'ов:

| Store | Что хранит |
|-------|-----------|
| `useAuthStore` | Пользователь, токены, статус авторизации |
| `usePlayerStore` | Текущий трек, очередь, громкость, прогресс |
| `useMusicStore` | Альбомы, треки, CRUD-операции |
| `useArtistStore` | Выбранный артист, открыта ли панель |
| `useLibraryStore` | Сохранённые альбомы + любимые треки |

`useLibraryStore` использует middleware **`persist`** — данные автоматически сохраняются в `localStorage` и не теряются при перезагрузке страницы.

### 6.2. Debounce при поиске

Чтобы не отправлять запрос на сервер при каждом нажатии клавиши, реализована задержка **400 миллисекунд** через `setTimeout` + `useRef`. Только если пользователь остановился — запрос уходит на сервер.

```typescript
debounceTimer.current = setTimeout(() => {
    doSearch(value);
}, 400);
```

Это стандартная практика, снижающая нагрузку на backend в несколько раз.

### 6.3. Исправление 307-редиректа (реальный баг в production)

В ходе разработки столкнулись с нетривиальным багом:

- FastAPI по умолчанию делает **307 редирект** с `/api/search` на `/api/search/` (без слэша → со слэшем)
- Nginx подставлял в заголовок `Location` **внутренний Docker-адрес** (`172.17.x.x:8000`), недоступный из браузера
- Браузер получал адрес и падал с `ERR_CONNECTION_REFUSED`
- Поиск молча не работал

**Решение** — двойной декоратор на эндпоинте, чтобы FastAPI принимал запросы и со слэшем, и без:

```python
@router.get("")   # принимает /api/search
@router.get("/")  # принимает /api/search/
def search(...):
    ...
```

## 8. Проблемы и пути их решения

За время разработки было выявлено и решено несколько реальных технических проблем:

| Проблема | Причина | Решение |
|---------|---------|---------|
| Поиск не работал | Конфликт глобального `isLoading` с параллельными запросами | Изолированный локальный `isSearching` в компоненте |
| 307-редирект в Docker | Nginx вставлял внутренний IP в `Location` | Двойной декоратор `@router.get("")` + `@router.get("/")` |
| 413 Payload Too Large | Nginx блокировал загрузку файлов | Увеличен лимит `client_max_body_size` до 50MB |
| Треки без обложки | При загрузке не подставлялась обложка альбома | Backend автоматически копирует `image_url` альбома в трек |
| ERR_CONNECTION_REFUSED | Неверный порт при обращении к приложению | Nginx маппит внутренний порт 80 на хостовый 3000 |
